Artifact 01

Chop Chop

A conversational AI meal-planning assistant.

Conversational AI Prompt Design Guardrails
Introduction

Chop Chop is a conversational AI assistant that plans a meal from the ingredients on hand, the number of people being fed, any dietary needs, and the time available to cook. It gathers missing details one question at a time rather than through a form, then returns a complete recipe: ingredient quantities scaled to the number of people, a full numbered method, prep and cook time, and a suggestion for what to serve alongside.

Objective

To design a conversational AI system from end to end: define its behaviour, test it against a range of realistic and adversarial inputs, and produce documented evidence that it performs as specified, rather than relying on a single successful demonstration.

Process

I began by iterating on the system prompt until Chop Chop consistently produced complete recipes rather than partial outlines, and asked a single clarifying question when information was missing instead of guessing at an answer.

I then built in explicit guardrails. A declared allergy is treated as a strict constraint rather than a preference. Requests for medical or clinical nutrition advice are declined, with the user directed to a professional instead. Off-topic requests, such as being asked to write an essay, are declined and redirected back to food.

I tested the system against a set of scenarios covering both core function and guardrail behaviour: a full recipe generated from a short ingredient list, a dietary substitution, a nut allergy, a deliberately vague prompt ("I'm hungry"), a request for medical advice, and an off-topic request. Each interaction was captured as a screenshot.

Tools

Claude, accessed via Cowork, for the build and iterative prompt design. Microsoft Word for the evidence documentation.

Value

This is a modest project, a meal planner rather than a commercial system, but the underlying skill is the one that transfers to more serious work: taking a loosely defined need and turning it into a system that behaves predictably, including at points where it would be easiest to improvise.

The guardrail behaviour is the part most worth noting. Producing a recipe from a model is straightforward. Deciding what the system must refuse to do, and demonstrating that it consistently does refuse, is what separates something suitable for a real user from something that merely performs well in a demonstration.

Evidence

Chop Chop runs in a Cowork sidebar rather than on a public server, so this is tested behaviour rather than a live link.

Artifact 02

Machine Learning vs Deep Learning

A comparative analysis built around one deciding question: does the problem have features a human can reasonably define by hand?

Linear Regression Convolutional Neural Networks Feature Engineering

Overview

Machine learning and deep learning are often used interchangeably, but treating them as the same thing obscures a distinction that determines which approach is suitable for a given problem. This report examines two examples: predicting football transfer value with linear regression as a machine learning (ML) case, and image recognition in autonomous driving as a deep learning (DL) case. The argument made is that the right choice between the two comes down to whether the problem has features a human can reasonably define by hand.

Case Study 1

Machine Learning – Predicting Football Transfer Value

Real-World Application: Player Market Valuation

Linear regression, commonly used for structured prediction problems such as house pricing, can also estimate a footballer's transfer value. Researchers built a multiple linear regression model to estimate the market value of forward players, using physical and performance factors such as age, goals scored, assists, and appearances across a season as inputs (Kologlu et al., 2018). This mirrors how clubs and agents already think about value informally – the model's task is to learn the weighting between these factors, rather than to discover which factors matter in the first place.

Why classical ML suits it
  • Countable, well-understood features. Goals, assists, age, and minutes are already available in structured form for every player in a league.
  • Minimal feature engineering. Nothing beyond selecting the right columns is required.
  • Interpretability. A club can point to a coefficient in a negotiation and argue that a player's goal return justifies a particular valuation.
Why deep learning does not
  • No hidden patterns. The inputs are already defined and few in number – there are no raw pixels or text for a neural network to discover structure in.
  • Overfitting risk. Training a many-layered model on a few thousand transfers would be as likely to overfit as to improve on the regression.
  • No upside. The regression is already interpretable and quick to train.
Case Study 2

Deep Learning – Image Recognition in Autonomous Driving

Real-World Application: Self-Driving Vehicles

A self-driving vehicle is typically built using deep reinforcement learning alongside convolutional neural networks (CNNs), and it is the CNN that performs the image recognition: processing a continuous stream of camera frames and LIDAR readings and recognising pedestrians, traffic lights, and road signs before the vehicle uses that understanding to navigate safely. This is a live, high-stakes application rather than a hypothetical one. Waymo was reported earlier this year to be operating fully autonomous ride-hailing across ten US metro areas, completing several hundred thousand paid rides per week, whilst incidents such as the January 2026 investigation into a Waymo vehicle near a Santa Monica school show just how much is riding on these perception systems getting it right (Dilmegani & Sezer, 2026).

Why deep learning suits it
  • No hand-writable feature list. There is no small, fixed set of features that could describe a pedestrian about to step into the road, or a traffic light obscured by glare.
  • Automatic feature learning. A camera frame is a grid of pixel values; a CNN learns its own hierarchy of features – edges, then shapes, then whole objects – directly from sensor input.
  • Learned driving policy. Reinforcement learning lets the system learn from the consequences of its actions rather than a predefined rulebook.
Why classical ML does not
  • Beyond hand-picked features. Even a well-tuned decision tree or regression, fed hand-picked image statistics, would struggle with the nonlinear, high-dimensional patterns in a road scene.
  • Brittle in the real world. It would likely fail outside narrow, controlled conditions – a shortcoming with obvious safety consequences at motorway speed.

The trade-off is that deep learning demands what ML does not: enormous labelled datasets drawn from millions of miles of driving, and substantial computing power to train and run in real time. That cost is justified here because the alternative would not solve the problem to any acceptable, or safe, standard.

Comparative Analysis

The deciding question – does the problem have features a human can reasonably define by hand? If it does, a simple, interpretable ML model is likely sufficient. If it does not, deep learning is the only realistic route, and its cost in data and computing power is the price of solving the problem at all.

Conclusion

These two examples show that the choice between machine learning and deep learning is not a question of which is better in the abstract, but of which matches the structure of the data in front of you. Predicting football transfer value succeeds with a simple, interpretable ML model precisely because its useful features are visible and easy to define by hand. Image recognition in autonomous driving requires deep learning precisely because its features are not, and can only realistically be discovered by a network with enough layers, sensor data, and computing power to learn them itself.

Recognising which situation applies is the real skill being tested by this comparison, rather than any inherent preference for one method over the other.
